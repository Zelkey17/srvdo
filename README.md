# srvdo

Одна команда для работы с сервером: команды, живучие сессии, файлы и порты — без запоминания флагов ssh, tmux и rsync.

```
srvdo init                     # настроить один раз
srvdo -t dev                   # живучая сессия на сервере
srvdo -d build 'make -j4'      # долгая задача в фоне
srvdo -w build                 # ждать и получить уведомление
srvdo watch ~/proj             # непрерывная синхронизация папки
srvdo tunnel                   # выбрать порт сервера и пробросить сюда
```

## Установка

Рекомендуемый способ — установщик. Он сам найдёт `~/.bashrc` и/или `~/.zshrc`, допишет туда одну строку `source` в отдельном блоке (легко найти и убрать), проверит зависимости и не тронет существующий файл без бэкапа:

```bash
curl -fsSL https://raw.githubusercontent.com/Zelkey17/srvdo/main/install.sh | bash
```

Или из клона репозитория:

```bash
git clone https://github.com/Zelkey17/srvdo
cd srvdo
./install.sh
```

Файл кладётся в `$XDG_DATA_HOME/srvdo/srvdo.sh` (обычно `~/.local/share/srvdo/srvdo.sh`), в rc-файл добавляется только:

```
# >>> srvdo >>>
[ -f "~/.local/share/srvdo/srvdo.sh" ] && . "~/.local/share/srvdo/srvdo.sh"
# <<< srvdo <<<
```

Полезные флаги: `--prefix=КАТАЛОГ` (куда класть файл), `--shell=bash,zsh` (в какие rc-файлы писать), `--no-rc` (только положить файл, ничего не дописывать), `--branch=ВЕТКА`. Полный список — `./install.sh --help`.

Без установщика — тем же способом, что и раньше:

```bash
mv srvdo.sh ~/srvdo.sh
echo 'source ~/srvdo.sh' >> ~/.bashrc
source ~/.bashrc
```

Дальше в любом случае:

```bash
srvdo init
```

`init` спросит алиас, адрес, пользователя, порт и рабочий каталог, затем:

1. создаст ключ `ed25519`, если его нет;
2. допишет блок `Host` в `~/.ssh/config` (существующий блок — только с вашего подтверждения, с бэкапом);
3. попросит пароль **один раз**, чтобы разложить ключ через `ssh-copy-id`;
4. предложит поставить на сервере `tmux` и `rsync`;
5. сохранит настройки в `~/.config/srvdo/config`.

Пароль никуда не сохраняется: вводится скрыто, при наличии `sshpass` передаётся через переменную окружения (не аргументом, чтобы не светиться в `ps aux`), и после раскладки ключа больше не нужен. Убедившись, что вход по ключу работает, стоит выключить на сервере `PasswordAuthentication`.

Необязательные, но полезные зависимости — `sudo apt install inotify-tools sshfs fzf`:

| пакет | что даёт |
|---|---|
| `inotify-tools` | `srvdo watch` — синхронизация по изменениям |
| `sshfs` | `srvdo mount` |
| `fzf` | `srvdo -t` и `srvdo tunnel` без аргументов предлагают выбор |

Установщик проверяет всё это сам и подсказывает команду для установки под ваш пакетный менеджер (`apt`, `dnf`, `pacman`, `brew`). Проверить ещё раз в любой момент: `srvdo doctor`.

### Обновление и удаление

Повторный запуск установщика (той же командой) обновляет `srvdo.sh` — предыдущая версия сохраняется рядом как `.bak` — и не плодит вторую строку в rc-файле, только пересобирает блок между метками.

```bash
curl -fsSL https://raw.githubusercontent.com/Zelkey17/srvdo/main/install.sh | bash -s -- --uninstall
# или, из клона:
./install.sh --uninstall
```

Уберёт строку из rc-файлов и удалит каталог с `srvdo.sh`; настройки (`~/.config/srvdo`) удалит только с подтверждением. Записи `Host` в `~/.ssh/config` и ключи не трогает — это доступ к вашему серверу, install.sh его не решает удалять сам.

## Общие опции

Ставятся **перед** командой и свободно сочетаются друг с другом:

```bash
srvdo --alias=abc -t axax              сессия axax на сервере abc
srvdo --alias=prod -n push ~/proj      репетиция отправки на prod
srvdo -q --no-reconnect -t build       без болтовни и без переподключений
srvdo --editor=vim edit /etc/hosts
srvdo --exclude='*.log' --exclude=tmp push ~/proj
```

| опция | что делает |
|---|---|
| `--alias=ИМЯ`, `--host=ИМЯ` | выполнить на другом сервере, не переключаясь насовсем |
| `--dir=ПУТЬ` | переопределить каталог назначения по умолчанию |
| `--editor=РЕД`, `--editor=?` | редактор для `edit` (`?` — выбрать из установленных) |
| `--peek=N` | сколько строк показывает `peek` |
| `--timeout=СЕК` | таймаут установки соединения |
| `--exclude=ШАБЛОН` | ещё одно исключение для rsync, можно повторять |
| `--config=ФАЙЛ` | другой файл настроек |
| `-n`, `--dry-run` | показать, что было бы сделано, ничего не меняя |
| `-y`, `--yes` | не спрашивать подтверждений |
| `-q`, `--quiet` | только ошибки |
| `--no-reconnect` | не переподключаться при обрыве |
| `--no-color` | без цвета (то же делает `NO_COLOR=1`) |

Опции действуют **только на этот вызов** — глобальные настройки не меняются. Форма `--opt значение` работает наравне с `--opt=значение`.

## Длинные и короткие формы

У каждой команды есть читаемое имя и короткий флаг — они равнозначны:

| длинно | коротко | | длинно | коротко |
|---|---|---|---|---|
| `session` | `-t` | | `kill` | `-k` |
| `run` | `-d` | | `rename` | `-r` |
| `peek` | `-p` | | `list` | `-l` |
| `attach` | `-a` | | `info` | `-i` |
| `send` | `-s` | | `follow` | `-f` |
| `wait` | `-w` | | `record` | `-L` |
| `detach` | `-D` | | | |

`srvdo run build 'make'` и `srvdo -d build 'make'` — одно и то же.

## Справка

```bash
srvdo -h            общая: опции, все команды по группам
srvdo edit ?        подробно про одну команду: смысл, примеры, подводные камни
srvdo help edit     то же самое
srvdo examples      готовые сценарии целиком
```

`?` работает после любой команды: `srvdo use ?`, `srvdo -t ?`, `srvdo tunnel ?`. Если в текущем каталоге есть файл с именем из одного символа, bash попытается раскрыть `?` как шаблон — тогда пишите `'?'` в кавычках или `srvdo help КОМАНДА`.

## Настройки

Файл `~/.config/srvdo/config`, показать — `srvdo config`, править — `srvdo config edit` (перечитывается сразу). Любую переменную можно переопределить на один вызов: `SRVDO_PEEK=300 srvdo -p web`.

| переменная | по умолчанию | смысл |
|---|---|---|
| `SRVDO_HOST` | `srv` | алиас из `~/.ssh/config` |
| `SRVDO_DIR` | `/root` | каталог на сервере по умолчанию |
| `SRVDO_KEY` | `~/.ssh/id_ed25519` | ключ, который создаёт `init` |
| `SRVDO_EDITOR` | пусто | редактор для `edit`; пусто → `$VISUAL`/`$EDITOR` → выбор |
| `SRVDO_PEEK` | `60` | сколько строк показывает `-p` |
| `SRVDO_MOUNT_BASE` | `~/mnt` | куда монтирует `mount` |
| `SRVDO_CACHE_TTL` | `30` | сек, кэш имён сессий для Tab |
| `SRVDO_PING_COUNT` | `4` | пакетов в `ping` |
| `SRVDO_CONNECT_TIMEOUT` | `8` | сек на установку соединения |
| `SRVDO_DEBOUNCE` | `0.4` | пауза перед `rsync` в `watch` |
| `SRVDO_RECONNECT` | `1` | переподключаться при обрыве `-t` |
| `SRVDO_CHECK_SPACE` | `1` | проверять свободное место перед `push` и `bak` |
| `SRVDO_EXCLUDE` | `.git node_modules …` | что не передаёт `rsync` |

Несколько серверов: `srvdo hosts` — список алиасов, `srvdo use prod` — переключить текущую оболочку, `srvdo use prod --save` — сделать основным, `srvdo ping prod` — проверить конкретный, не переключаясь.

## Команды

### Разово

```bash
srvdo                          # интерактивный шелл
srvdo ls -la /srv              # выполнить и вернуть вывод сюда
srvdo apt-get install -y htop
```

### Сессии tmux

Сессия — это «комната» на сервере, которая живёт независимо от вашего подключения. Закрыли ноутбук — процессы внутри продолжают работать.

```bash
srvdo -t AAA                   # создать AAA или подключиться, если есть
srvdo -t AAA 'npm run dev'     # то же + запустить команду при создании
srvdo -t                       # выбрать из существующих (нужен fzf)
srvdo -t --list                # таблица: имя, окна, подключена ли, когда создана
srvdo -t --last                # вернуться в последнюю использованную
srvdo -k AAA                   # удалить
srvdo -k --all                 # удалить все (спросит подтверждение)
srvdo detach AAA               # отцепиться снаружи, сессия работает
srvdo -r AAA BBB               # переименовать
```

#### Выйти, не закрывая сессию

Это три разные вещи, и их легко перепутать:

| что нужно | как |
|---|---|
| выйти, оставив всё работать | `Ctrl-b d` изнутри, или `F12` после `srvdo keys` |
| отцепиться снаружи | `srvdo detach AAA`, `srvdo detach --all` |
| закрыть сессию вместе с процессами | `srvdo kill AAA` |

`srvdo keys` дописывает на сервере в `~/.tmux.conf` отдельный блок (с бэкапом): `F12` и `Alt-d` выходят из сессии без всяких префиксов, включается прокрутка мышью, история поднимается до 50000 строк. Смысл в том, что стандартный `Ctrl-b d` конфликтует с локальной сессией, если вы держите её и у себя — а `F12` уходит напрямую в дальнюю. Уже открытые сессии подхватят настройки после `srvdo 'tmux source ~/.tmux.conf'`.

`detach` нужен для случая, когда сессия осталась подключённой в закрытом терминале или на другой машине — процессы внутри при этом не трогаются.

При обрыве связи `-t` переподключается сам (до `SRVDO_RECONNECT_MAX` попыток) — полезно на нестабильном вайфае.

### Наблюдение за сессией, не входя в неё

```bash
srvdo -p AAA                   # показать последние строки вывода и выйти
srvdo -a AAA                   # смотреть живьём, только чтение
srvdo -s AAA "текст"           # отправить строку, как будто напечатали
srvdo -L AAA                   # писать вывод в файл на сервере
srvdo logs AAA                 # читать этот файл (tail -f)
```

`-s` работает через `tmux send-keys`, то есть буквально эмулирует клавиатуру. Если в этот момент сессия не ждала ввода, строка уйдёт не туда — проверяйте результат через `-p`.

### Фоновые задачи с уведомлением

```bash
srvdo -d build 'make -j4'      # запустить в фоне
srvdo -w build                 # ждать завершения
```

`-d` заворачивает команду так, чтобы код возврата сохранился на сервере, поэтому `-w` знает, чем всё кончилось:

| результат | уведомление |
|---|---|
| код 0 | `--urgency=low`, «успешно за Nс» |
| код ≠ 0 | `--urgency=critical`, «код N» |
| сессия закрыта не через `-d` | `--urgency=normal`, код неизвестен |

`Ctrl-C` во время `-w` прекращает ожидание, но не задачу.

### Файлы

```bash
srvdo diff ~/proj              # что уедет и что удалится — ничего не меняя
srvdo push ~/proj              # залить
srvdo push ~/proj/ /srv/app    # слэш на конце: содержимое папки, а не сама папка
srvdo pull /root/results.csv   # забрать
srvdo watch ~/proj             # держать синхронизированным непрерывно
srvdo edit /etc/nginx/nginx.conf   # скачать, открыть в $EDITOR, вернуть
srvdo bak /srv/data            # забрать каталог сюда одним tar.gz
srvdo mount                    # $SRVDO_DIR через sshfs в ~/mnt/АЛИАС
srvdo umount
```

Про слэш: `push ~/proj` создаст `/root/proj`, а `push ~/proj/` положит файлы прямо в `/root`. Это поведение самого rsync, а не выдумка обёртки. `diff` показывает результат заранее — привычка запускать его перед `push` спасает от неожиданных удалений, потому что `watch` и `diff` работают с `--delete`.

### Выбор редактора

`edit` ищет редактор в таком порядке: `--editor=` → `SRVDO_EDITOR` → `$VISUAL` → `$EDITOR` → интерактивный выбор из установленных (`nvim`, `vim`, `micro`, `nano`, `helix`, `emacs`, `kak`, `code`, `codium`, `subl`, `gedit`, `kate`). Выбор предложит запомнить в конфиге. Принудительно вызвать меню — `--editor=?`.

Для графических редакторов (`code`, `codium`, `subl`) автоматически добавляется `--wait`, иначе они возвращают управление сразу и файл отправился бы неотредактированным.

`edit` перед перезаписью оставляет на сервере `.srvdo-bak`, а при неудачной отправке сообщает, где лежит ваша правка. С `-n` покажет обычный `diff` вместо отправки.

### Сеть и состояние

```bash
srvdo flush                    # сбросить зависшие ssh-соединения
srvdo flush --all              # закрыть соединения со всеми серверами
srvdo ping                     # ICMP + порт + ssh-рукопожатие
srvdo -i                       # uptime, диск, память, сессии
srvdo ports                    # какие порты слушает сервер
srvdo tunnel 8080              # localhost:8080 → сервер:8080
srvdo tunnel 3000:8080         # localhost:3000 → сервер:8080
srvdo tunnel                   # выбрать порт из живых (нужен fzf)
srvdo -f /var/log/syslog       # tail -f
srvdo doctor                   # что установлено здесь и там
```

## Зависшие соединения

`ControlMaster` держит одно ssh-соединение и переиспользует его — за счёт этого команды и Tab работают мгновенно. Но если сеть оборвалась некрасиво, файл сокета остаётся, а мастер-процесс уже мёртв, и тогда любая команда молча ждёт таймаута вместо переподключения.

`srvdo flush` закрывает мастер-соединение через `ssh -O exit`, удаляет сокеты, не отвечающие на `ssh -O check`, и сбрасывает кэш подсказок. Живые соединения не трогает. Имя сокета в шаблоне из `init` (`cm-user@host:port`) позволяет проверить каждый по отдельности.

Признак, что пора: команды подвисают на десятки секунд, хотя `srvdo ping` показывает открытый порт и принятый ключ.

## Прерывание длинных операций

`watch`, `tunnel` и `mount` ставят обработчик `SIGINT`/`SIGTERM` только на время работы и снимают его после, восстанавливая прежний, если он был. Раньше `Ctrl-C` мог оставить точку монтирования занятой наполовину или прервать `watch` без сообщения.

| операция | что делает по `Ctrl-C` |
|---|---|
| `watch` | завершает цикл и сообщает, сколько прогонов синхронизации прошло |
| `tunnel` | печатает, что туннель закрыт; отличает обрыв от «порт не слушается» |
| `mount` | освобождает точку монтирования и убирает пустой каталог |

## Проверка свободного места

Перед `push` считается размер источника (с учётом исключений) и сравнивается со свободным местом на сервере; перед `bak` — размер каталога на сервере против свободного места здесь. Если не хватает — предупреждение с числами и вопрос, продолжать ли; если останется меньше двойного запаса — просто предупреждение.

Обе оценки сознательно завышены: `rsync` пошлёт только изменившееся, а `tar.gz` сожмётся. Поэтому это подсказка, а не запрет — `-y` соглашается сразу, `SRVDO_CHECK_SPACE=0` отключает проверку совсем (полезно, если `du` по большому каталогу на сервере работает долго).

## Когда что-то не работает

`srvdo ping` проверяет три слоя по очереди, и место обрыва прямо указывает на причину:

| ICMP | порт | SSH | что это значит |
|:---:|:---:|:---:|---|
| ✗ | ✗ | — | сервер выключен, адрес неверный или всё режет файрвол |
| ✗ | ✓ | ✓ | всё в порядке: ICMP просто закрыт у провайдера, это нормально |
| ✓ | ✗ | — | сервер жив, но ssh не слушает этот порт — проверьте `Port` в `srvdo config` |
| ✓ | ✓ | ✗ | сеть в порядке, дело в доступе: ключ не разложен или не тот пользователь → `srvdo init` |

Прочие частые сообщения:

| что видите | что делать |
|---|---|
| `не удалось подключиться` + список проверок | пройдите по списку; он подставляет ваши реальные значения |
| `на сервере не установлен tmux` | `srvdo apt-get install -y tmux` |
| `в имени сессии нельзя ':' и '.'` | ограничение tmux; в сообщении есть исправленный вариант |
| `сессии «X» нет` + перечень живых | опечатка в имени, либо создайте: `srvdo -t X <команда>` |
| `сессия «X» уже работает` | `-t` подключается к существующей и не запускает команду повторно. Нужен чистый старт — `srvdo -k X` |
| `часть файлов не передалась` (rsync 23) | нет прав на каталог назначения на сервере |
| `на сервере нет rsync` (rsync 12) | `srvdo apt-get install -y rsync` |
| `локальный порт занят` | `srvdo tunnel ДРУГОЙ:УДАЛЁННЫЙ` |
| команды подвисают на 10–30 с | `srvdo flush` — вероятно, мёртвый мастер-сокет |
| Tab-дополнение тормозит | добавьте `ControlMaster auto` в `~/.ssh/config`; `srvdo doctor` это проверяет |

## Подсказки по Tab

Дополнение зависит от позиции, а не от одного предыдущего слова:

| где нажали Tab | что предложит |
|---|---|
| `srvdo <Tab>` | только длинные имена команд |
| `srvdo -<Tab>` | общие опции (короткие флаги команд в подсказках не показываются) |
| `srvdo --alias=<Tab>` | алиасы из `~/.ssh/config` |
| `srvdo -t <Tab>`, `srvdo session <Tab>` | имена живых сессий, `--list`, `--last` |
| `srvdo -k <Tab>`, `srvdo detach <Tab>` | имена сессий и `--all` |
| `srvdo use <Tab>` / `srvdo use prod <Tab>` | алиасы / `--save` |
| `srvdo push <Tab>` | локальные файлы |
| `srvdo pull <Tab>`, `srvdo edit <Tab>` | пути на сервере |
| `srvdo push ~/x <Tab>` | пути на сервере (второй аргумент — назначение) |
| `srvdo keys <Tab>`, `srvdo doctor <Tab>`, `srvdo init <Tab>` | ничего: аргументов не принимают |
| `srvdo -t AAA <Tab>`, `srvdo -d job <Tab>` | исполняемые файлы **сервера** |
| `srvdo -t AAA rm <Tab>`, `srvdo rm <Tab>` | пути **на сервере** |

Опции распознаются на любой позиции перед командой, поэтому `srvdo --alias=prod -n -q kill <Tab>` всё равно предложит сессии.

Слова собираются из `COMP_LINE` вручную, а не берутся из `COMP_WORDS`: bash режет слово по символам `COMP_WORDBREAKS` (`=`, `:` и прочие), поэтому `--alias=prod` приходит туда как три отдельных слова и позиция команды считалась бы неправильно. Готовые варианты затем обрезаются до последнего такого разделителя — иначе `--alias=<Tab>` вставлял бы `--alias==prod` с двойным знаком равенства.

Пути на сервере раскрывает сам сервер, поэтому подставляются реальные файлы, а каталоги помечаются слэшем. Префикс уходит туда через stdin, а не подставляется в текст команды: иначе имена с пробелами и кавычками не находились бы, а специально подобранный префикс мог бы вырваться из кавычек и выполнить постороннюю команду. Список исполняемых файлов сервера кэшируется на час, имена сессий кэшируются на `SRVDO_CACHE_TTL` секунд в `/tmp/.srvdo-АЛИАС-UID`. Кэш перезаписывается только при успешном ответе сервера, а при чтении строки фильтруются — иначе одна строка с пробелами превращалась бы в несколько мусорных вариантов. Сбросить вручную: `srvdo cache-clear`. `srvdo list` тоже освежает кэш.

## Как устроено

- Все удалённые вызовы идут через одну обёртку, которая отличает ошибку соединения (ssh возвращает 255) от ошибки самой команды и в первом случае печатает разбор причин.
- Имена сессий и путей уходят на сервер через `printf %q`, а команды для `-d` — в base64. Поэтому кавычки, пробелы и кириллица в аргументах не ломают ничего.
- `-t` использует `tmux new-session -A`: одна и та же команда и создаёт сессию, и подключается к существующей.
- `-k` использует `kill-session -t=ИМЯ` со знаком равенства — точное совпадение. Без него tmux допускает совпадение по префиксу, и `-k AA` снёс бы `AAA`.
- Имена сессий для Tab кэшируются на `SRVDO_CACHE_TTL` секунд, чтобы не дёргать ssh на каждое нажатие.

## Zsh

Работает и в zsh — установщик сам находит `~/.zshrc` (или `$ZDOTDIR/.zshrc`) и дописывает туда ту же строку `source`, что и в `~/.bashrc`; вручную ничего добавлять не нужно. Что для этого сделано внутри `srvdo.sh`:

- ввод (`init`, подтверждения, пароль) идёт через обёртки, потому что `read -p` и `read -s` в zsh значат совсем другое;
- вместо `mapfile` используется переносимый цикл чтения;
- в дополнении нет числовых индексов массивов: zsh нумерует их с единицы;
- подстроки записаны как `${var:0:$n}` — без `$` zsh считает это модификатором истории и падает;
- дополнение подключается через `bashcompinit`, который умеет исполнять bash-функции дополнения.

Отдельно про `use`: в zsh нельзя изменить глобальную переменную из функции, где объявлена её локальная тень (ни `declare -g`, ни `typeset -g`, ни `unset` не помогают). Поэтому эффективный сервер хранится во внутренней переменной, а `SRVDO_HOST` локальной не делается — иначе `srvdo use prod` молча не срабатывал бы.

## Ограничения

- Bash 4.3+ или zsh 5.x.
- `umount` использует `fusermount` — это Linux; на macOS нужно `umount`.
- Установка пакетов в `init` умеет `apt-get`, `dnf` и `apk`; на прочих системах поставьте `tmux` и `rsync` вручную.
- Синхронизация `watch` односторонняя (локально → сервер) и с `--delete`. Для двусторонней смотрите Syncthing или Unison.
